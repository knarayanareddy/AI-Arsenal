## What & Why

Include adversarial and malformed inputs (contradictory instructions, boundary-pushing requests, malformed data) in your prompt eval set, not only clean, well-formed test cases. A prompt validated only against typical inputs can fail unpredictably on inputs that deviate from that pattern, and this gap is invisible until specifically tested for.

## Before / After

**Before:** eval set consists entirely of clean, well-formed, typical user inputs.

**After:** eval set includes a subset of adversarial cases (contradictory requests, boundary values, malformed structure) alongside the typical cases, with explicit pass/fail criteria for each.

## Implementation

Identify realistic adversarial or edge-case input patterns specific to your application (not a generic list), add a representative sample to your eval set with defined expected behavior, and run them alongside typical-case evaluation.

## Gotchas

- A prompt validated only on well-formed inputs can fail unpredictably on adversarial ones — this gap is invisible until specifically tested
- Adversarial test inputs need to reflect realistic behavior for your specific application, not a generic unrelated list

## When NOT to Apply

- Skip this for tightly-constrained inputs that are structurally guaranteed to be well-formed (e.g. generated by another trusted internal system with strict validation)
- Not a substitute for `run-prompt-injection-regression-tests` when the actual concern is security rather than general robustness — use both for a security-sensitive application
