# Observability Vertical Reorganisation — Migration Completion Report

**Scope:** Reorganise `content/observability/` (4 flat entries under `entry_type: "guide"` + `section: "observability"`) into 7 category folders (`instrumentation/`, `tracing/`, `evaluation-quality/`, `monitoring-alerting/`, `cost-usage/`, `privacy-governance/`, `incident-response/`), introducing a new dedicated `observability` entry type with `category`, `scope`, `signal_types[]`, `verification_status`, `data_sensitivity[]`, and a required `instrumentation_contract` object (events, sampling, retention, correlation, redaction) as new required schema fields, and enforcing a canonical, STRICTLY-ORDERED 10-section body format with automated Failure-Mode-1/2/3 content checks — the first vertical in this catalog to enforce exact section ordering, not just presence.

## Why this vertical was different (and how that shaped the work)

Per the brief's own framing, this vertical answers "how do I KNOW what the system is doing in prod, and how do I debug/improve it without guessing?" — a categorically operational question, distinct from every other vertical's decision-support or build-artifact framing. The brief names three specific failure modes (vendor pages disguised as guidance, abstract advice with no instrumentation contract, PII-unsafe suggestions) and — critically — includes an explicit "WHAT YOU MUST NEVER DO" section with hard prohibitions (never claim production-verified without evidence, never recommend unsafe logging, never leave the instrumentation contract vague). This migration treated every one of these as an automatable, non-negotiable content-quality gate, not editorial guidance: `has_decision_framework`-style re-verification for `verification_status: production-verified` claims, an unsafe-logging heuristic that flags the literal failure mode named in the brief, and — new to this vertical specifically — **strict section-order enforcement**, since the brief states "EXACT order, do not reorder/rename" where every prior vertical only checked heading presence.

The single largest finding from the Phase 1 audit: **zero pre-existing observability content contained any privacy/PII/redaction/retention guidance**, despite the pre-existing `overview.md` and `cost-tracking.md` entries explicitly discussing capturing prompts, tool outputs, and user/tenant IDs into traces. This is the brief's own named Failure Mode 3 present by omission in every single pre-existing entry — not a hypothetical risk the brief was guarding against preemptively, but an actual, pre-existing gap this migration closed with a dedicated `privacy-governance/redact-and-govern-trace-data.md` entry that every content-capturing entry now cross-links to rather than re-describing redaction independently.

## 2. Instrumentation Engineer — Evidence & Implementation Reality

Applied the Six Questions to every entry during authoring, with a live-research policy applied specifically to vendor-specific and standards-body claims:

- **Live-verified, not invented, instrumentation contracts.** Every entry's field names and event shapes are grounded in live web research conducted during authoring, not fabricated: `capture-the-llm-call-event.md`'s `gen_ai.client.inference.operation.details` event and the content-capture-off-by-default posture come directly from current (2026) OpenTelemetry GenAI Semantic Conventions SIG documentation; `trace-every-agent-and-rag-step.md`'s `invoke_agent`/`execute_tool` span conventions likewise; `redact-and-govern-trace-data.md`'s gateway-boundary-before-persistence pattern and Microsoft Presidio-based implementation are grounded in multiple independent 2026 practitioner sources on PII redaction for LLM systems; `alert-on-quality-and-cost-regressions.md`'s multi-window burn-rate alerting thresholds (14.4x/1hr, 3x/6hr) follow the standard SRE burn-rate pattern, explicitly cited from a 2026 LLM-monitoring source in its Resources section.
- **Honest `verification_status` assignment, not defaulted to `production-verified` across the board.** 5 of 7 entries (`capture-the-llm-call-event.md`, `trace-every-agent-and-rag-step.md`, `gate-releases-on-eval-regression.md`, `instrument-cost-per-feature.md`, `redact-and-govern-trace-data.md`) carry `production-verified` with an explicit evidence-type description per the Validator's rule (see below) — the underlying patterns are standard, widely documented, and corroborated by multiple named tools/standards bodies in this catalog. The remaining 2 of 7 (`alert-on-quality-and-cost-regressions.md`, `runbook-for-agent-and-rag-incidents.md`) are honestly downgraded to `community-reported`, each with an `enrichment_notes` field explaining precisely why: the underlying patterns (burn-rate alerting, incident kill-switches) are standard SRE practice, but their specific application to AI-system signals hasn't been independently confirmed against one named production system by an Arsenal maintainer.
- **Real code/config in every Implementation section**, not pseudocode: working Python snippets for span instrumentation, cost computation, redaction (using Presidio), CI eval-gating, and a feature-flag-backed kill switch.

## 4. Validator — Schema & Taxonomy Enforcement

- **Three real defects found and fixed during authoring**, each caught by actually running the new checks against real content, not just unit-testing them in isolation:
  1. `trace-every-agent-and-rag-step.md`'s Privacy & Governance retention regex check (`retention|delet\w*|expir\w*|ttl\b`) failed to match the word "retained," incorrectly flagging a genuinely compliant retention paragraph. Fixed by adding `retain\w*` to the pattern, re-verified against both the originally-failing entry and the first entry (to confirm no regression).
  2. Every one of the first 5 entries authored omitted the catalog-wide `added_date`/`added_by`/`status` fields (required by every entry type in this repository, though not shown in the brief's own YAML template example) — caught immediately by schema validation on each entry's first save, fixed each time.
  3. `alert-on-quality-and-cost-regressions.md`'s honest `community-reported` verification status needed an `enrichment_notes` field to explain the distinction from `production-verified`, matching a pattern used by every other vertical's schema but not originally specified in this vertical's Step 1.5 field list — added as a legitimate additive schema extension (`schemas/observability.schema.json` v1.0.0, `enrichment_notes` optional field) rather than omitting the explanation.
- **A real bug in the migration guard script's own arithmetic**, found and fixed after the final retirement: `check-observability-migration-progress.js` initially computed `total = allObsFiles.length - RETIRED_LEGACY_IDS.size`, which was only correct while the retired file (`overview.md`) still physically existed on disk awaiting deletion. Once actually deleted (the normal end state), this subtraction double-counted the retirement and produced an inflated 116.7% completion figure. Fixed by deriving `total` from `migrated.length + pending.length` instead — both of which already correctly exclude retired entries via the early `continue` — removing the class of off-by-one regardless of whether the retired file has been deleted yet.
- **The strict-order heading check (new to this vertical) works correctly**: verified across all 7 entries with no false positives, confirming `STRICT_ORDER_TYPES` correctly gates this behavior to observability entries only, leaving every other vertical's presence-only check unaffected.
- Final verification: `pnpm test` (131/131 pass), `validate:schema`/`validate:taxonomy`/`validate:structure`/`validate:paths`/`validate:refs`/`check:duplicates` all pass on 374 total entries, all six migration-guarded verticals report 100%, and `generate:all` regenerates the full data layer (`data/observability.json`, search index with new `scope`/`signal_types`/`data_sensitivity` facets, category registries, `CONTEXT.md`'s new "Observability Playbooks by Category" section, stats) cleanly.
